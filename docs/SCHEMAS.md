# SCHEMAS.md — Data Schema Reference

## Overview
This document defines the **contract schemas** used throughout the Learning Helper system.

### Source of Truth (Hard Rule)
- **Source of truth:** Pydantic v2 models in the Python Curriculum service: `apps/curriculum/src/models/`
- **Generated artifacts:** JSON Schema files exported to `packages/contracts/schemas/` (generated-only; never hand edit)
- **Consumers:**
  - **Node Orchestrator** validates Python responses using **AJV** against the generated JSON Schemas
  - **Node Orchestrator** uses **Zod** for **public API input** validation only (user → Node boundary)

### Contract Versioning (Hard Rule)
Every payload emitted by the Python service MUST include:
- `schema_version` (e.g., `"plan.v1"`, `"exercise_set.v1"`)
- `metadata` including `provider`, `model`, `prompt_version`, `created_at`, plus audit fields listed below

Breaking changes require a new schema version file and backward-compat support during migration.

---

## Identity & Scoping Rules (Non-Negotiable)

### Plan Scope
- `plan_id` is globally unique (UUID) and assigned/persisted by the Node Orchestrator.
- `node_id` is **plan-scoped** (unique only within a plan). It MUST NOT be treated as globally unique.

### Required Key Strategy
- Persist and reference nodes using **(plan_id, node_id)** as the logical key.
- Any table/entity that references a node MUST carry `plan_id` and `node_id`.

---

## Shared Metadata Schema

### ArtifactMetadata (used by all Python-emitted artifacts)
**Purpose:** Enable auditing and reproducibility.

Required fields:
- `provider`: `"gemini" | "claude" | "<other>"`
- `model`: provider model identifier
- `prompt_version`: e.g., `"plan/v2"`
- `created_at`: ISO-8601 datetime
- `request_id`: UUID generated by Node and propagated to Python
- `raw_output_hash`: SHA-256 hash of the raw provider response string
- `artifact_hash`: SHA-256 hash of the validated JSON artifact (canonicalized JSON)
- `validation_retry_count`: integer (0–2)

Notes:
- Do **not** include secrets or full raw outputs in metadata.
- The Node service persists metadata for every operation.

## Plan Schema (Python Output)

### Plan (plan.v1)
A complete learning roadmap for a topic.

Top-level requirements:
- `schema_version`: `"plan.v1"`
- `topic`: 3-100 chars
- `user_level`: `"beginner" | "intermediate" | "advanced"`
- `plan_size`: `"basic" | "moderate" | "large" | "dynamic"` (default: `"moderate"`)
- `nodes`: node count depends on `plan_size`:
  - `"basic"`: 4-12 nodes (quick overview)
  - `"moderate"`: 12-20 nodes (standard depth)
  - `"large"`: 20-30 nodes (comprehensive)
  - `"dynamic"`: 4-30 nodes (LLM decides based on topic)
- `schedule`: covers all nodes exactly once; sequential order starting at 1
- `metadata`: `ArtifactMetadata`

#### Node
A single learning unit within a plan.

Constraints:
- `node_id`: snake_case, `[a-z0-9_]+`, 3–100 chars, unique within plan
- `title`: 5–200 chars
- `objectives`: 1–5 non-empty strings
- `prerequisites`: list of plan-scoped `node_id` values
- `estimated_minutes`: 5–240
- `tags`: optional list of strings

Hard rules:
- A node may not list itself as a prerequisite.
- Node prerequisite cycles are rejected by Node semantic validation (no auto-repair).

#### ScheduleItem
Defines plan order (linear learning path) while prerequisites remain a DAG.

Constraints:
- `order`: sequential integers starting from 1
- `node_id`: must reference an existing node in `nodes`
- All nodes must appear exactly once in `schedule`

## Exercise Schema (Python Output)

### ExerciseSet (exercise_set.v1)
A collection of exercises for a specific node in a specific plan.

Top-level requirements:
- `schema_version`: `"exercise_set.v1"`
- `plan_id`: UUID (echoed back from Node request)
- `node_id`: plan-scoped node id
- `user_level`: `"beginner" | "intermediate" | "advanced"`
- `exercises`: 1+ exercises
- `metadata`: `ArtifactMetadata`

Hard rule:
- Exercise IDs must be unique within the returned `ExerciseSet`.

## Exercise Types (Discriminated Union)

### Exercise (exercise.v1)
`Exercise` MUST be a discriminated union keyed by `type`.
All variants share:
- `id`: string (recommended: UUID or `ex_<type>_<shortid>`)
- `type`: one of `"mcq" | "short_answer" | "fill_blank" | "coding" | "flashcard"`
- `prompt`: min 10 chars
- `rubric`: 20–500 chars
- `difficulty`: integer 1–5

#### MCQExercise
- `type`: `"mcq"`
- `choices`: exactly 4 strings (min 1 char each)
- `correct_answer`: string that MUST match one of `choices`

#### ShortAnswerExercise
- `type`: `"short_answer"`
- `correct_answer`: string (reference answer; grader accepts semantically equivalent answers)

#### FillBlankExercise
Supports one or multiple acceptable answers.

- `type`: `"fill_blank"`
- `correct_answer`: object:
  - `answers`: list of acceptable strings (min 1, max 10)
  - `match`: `"case_sensitive" | "case_insensitive"`
  - `normalize_whitespace`: boolean

Rationale:
- Avoids contradictions between “exact match” and “acceptable variations”.

#### CodingExercise
- `type`: `"coding"`
- `correct_answer`: object:
  - `language`: string (e.g., `"python"`)
  - `solution`: string
  - `test_cases`: list of `{ input: any, output: any }` (min 1, max 20)

Hard rules:
- The Python service MUST NOT claim execution results.
- Node may optionally run tests in a sandbox post-MVP; MVP grading is rubric-based.

#### FlashcardExercise
- `type`: `"flashcard"`
- `correct_answer`: string (expected concept statement)

## Grade Schema (Python Output)

### Grade (grade.v1)
The result of grading a user's answer.

Top-level requirements:
- `schema_version`: `"grade.v1"`
- `plan_id`: UUID (echoed back from Node request)
- `node_id`: plan-scoped node id (echoed back from Node request)
- `exercise_id`: string
- `score`: 0.0–1.0
- `is_correct`: boolean
- `feedback`: 20–300 chars
- `misconceptions`: optional list of strings
- `metadata`: `ArtifactMetadata`

Hard rule for correctness:
- `is_correct` MUST be `true` if `score >= 0.7`
- `is_correct` MUST be `false` if `score < 0.5`
- For `0.5 <= score < 0.7`, `is_correct` may be either based on rubric intent (but should be consistent).

## Query Suggestions Schema (Python Output)

### QuerySuggestions (query_suggestions.v1)
Suggested YouTube search queries for a node.

Top-level requirements:
- `schema_version`: `"query_suggestions.v1"`
- `plan_id`: UUID (echoed back from Node request)
- `node_id`: plan-scoped node id
- `queries`: 1–5 strings (min length 3)
- `metadata`: `ArtifactMetadata`

Hard rules:
- Queries are plain strings only.
- No URLs, no video IDs, no channel names, no “this video teaches X” claims.

## YouTube Resource Schemas (Node-Owned, Not LLM)
YouTube resources are **deterministic artifacts** produced by the Node Orchestrator via the YouTube Data API.
These schemas are NOT produced by LLM output and MUST NOT appear in Python LLM output schemas.

Recommended Node domain shape (not exported from Pydantic):
- `video_id` (11-char YouTube ID)
- `title`
- `channel_title`
- `duration_seconds`
- `rank_score` (0–1)
- `type`: `"must_watch" | "recommended"`
- `rationale` (<= 240 chars)
- `url` should be derived from `video_id` by Node (`https://www.youtube.com/watch?v=${video_id}`)

## Plan Caching Schemas

### Overview
Plan caching uses a **hybrid Postgres + Redis** approach:
- **Postgres**: Source of truth for cached plans, quality signals, and audit history
- **Redis**: Hot-path acceleration for popular topics (read-through cache)

### Postgres: `cached_plans` Table
Stores validated learning plans keyed by normalized topic + user level.

| Column | Type | Description |
|--------|------|-------------|
| `cache_key` | `VARCHAR(64) PRIMARY KEY` | SHA-256 of normalized topic + user_level |
| `topic_raw` | `VARCHAR(500)` | Original user input (before normalization) |
| `topic_normalized` | `VARCHAR(500)` | LLM-normalized canonical topic |
| `user_level` | `VARCHAR(50)` | beginner / intermediate / advanced |
| `plan_data` | `JSONB` | Full validated Plan artifact (all mastery levels) |
| `domain_category` | `VARCHAR(100)` | e.g., "networking/protocols", "math/calculus" |
| `staleness_policy` | `VARCHAR(50)` | "7d", "30d", "90d", "annual", "never" |
| `created_at` | `TIMESTAMP` | When plan was first generated |
| `last_staleness_check` | `TIMESTAMP` | Last MCP freshness verification |
| `staleness_check_result` | `JSONB` | MCP response with sources checked |
| `invalidated_at` | `TIMESTAMP` | Soft delete timestamp (NULL if valid) |
| `invalidation_reason` | `VARCHAR(255)` | "low_completion_rate", "stale_content", etc. |

Indexes:
- `idx_cached_plans_domain` on `domain_category`
- `idx_cached_plans_created` on `created_at`

### Postgres: `plan_quality_snapshots` Table
Tracks quality signals over time for trend analysis and regeneration decisions.

| Column | Type | Description |
|--------|------|-------------|
| `cache_key` | `VARCHAR(64)` | FK to cached_plans |
| `snapshot_date` | `DATE` | Aggregation date |
| `sample_size` | `INT` | Number of users in sample |
| `completion_rate` | `FLOAT` | % of users completing >60% of nodes |
| `avg_time_to_mastery_minutes` | `FLOAT` | Actual vs. estimated time |
| `exercise_pass_rate` | `FLOAT` | First-attempt pass rate |
| `resource_engagement_rate` | `FLOAT` | % of users engaging with videos |
| `negative_feedback_rate` | `FLOAT` | Explicit thumbs-down rate |

Primary key: `(cache_key, snapshot_date)`

### Redis: Cache Key Conventions

Hot plan cache (read-through from Postgres):
```
plan_cache:{cache_key}
TTL: 24 hours (configurable via PLAN_CACHE_HOT_TTL_SECONDS)
Value: JSON string of plan_data
```

MCP staleness check cache (avoid repeated checks):
```
staleness_check:{cache_key}:{date}
TTL: 24 hours
Value: JSON { "stale": boolean, "reason": string?, "sources_checked": string[] }
```

### Quality Signal Thresholds (Defaults)

Regeneration is triggered when **any** of these conditions are met (and sample_size >= 20):

| Signal | Threshold | Rationale |
|--------|-----------|-----------|
| `completion_rate` | < 0.50 | Half of users abandon the plan |
| `exercise_pass_rate` | < 0.40 | Exercises too hard or unclear |
| `negative_feedback_rate` | > 0.15 | Explicit user dissatisfaction |
| `avg_time / estimated_time` | > 2.0 | Time estimates are significantly wrong |

> [!NOTE]
> These thresholds are initial defaults. Future work may replace them with a trained model.

## Schema Generation & Consumption

### Python (Generate JSON Schema)
- Python service exports JSON Schemas from Pydantic v2 models into:
  - `packages/contracts/schemas/*.schema.json`

Generation output MUST be deterministic and committed to the repository.

### Node (Validate Python Responses)
- Node service uses **AJV** validators loaded from `packages/contracts/schemas/`.
- Node rejects any Python response that fails schema validation (no coercion).

---

## Migration Rules

When a breaking schema change is needed:
1. Create a new schema version (e.g., `plan.v2.schema.json`)
2. Update Python to emit the new `schema_version`
3. Update Node to accept both versions during migration
4. Deprecate and remove old versions after migration window

## Transcript and Validation Schemas (Python Output)

### Transcript (transcript.v1)
Raw transcript data fetched from YouTube videos.

Top-level requirements:
- `schema_version`: `"transcript.v1"`
- `video_id`: 11-character YouTube video ID
- `language`: language code (default: `"en"`)
- `segments`: array of timed transcript segments
- `full_text`: concatenated transcript text
- `duration_seconds`: total video duration
- `fetch_source`: `"youtube_transcript_api" | "youtube_api" | "manual"`

#### TranscriptSegment
- `start_seconds`: segment start time (>= 0)
- `duration_seconds`: segment duration (>= 0)
- `text`: segment text (min 1 char)

---

### VideoValidation (video_validation.v1)
Result of LLM validation comparing a video transcript against a learning node.

Top-level requirements:
- `schema_version`: `"video_validation.v1"`
- `video_id`: 11-character YouTube video ID
- `plan_id`: UUID (echoed from request)
- `node_id`: plan-scoped node id
- `is_relevant`: boolean
- `relevance_score`: 0.0–1.0
- `matched_objectives`: list of matching objective strings
- `rejection_reason`: optional explanation if not relevant
- `metadata`: `ArtifactMetadata`

Relevance thresholds:
- Videos with `relevance_score >= 0.6` are considered relevant
- Videos with `relevance_score < 0.6` are filtered out

---

### StalenessResult (staleness_result.v1)
Result of checking cached content against current sources.

Top-level requirements:
- `schema_version`: `"staleness_result.v1"`
- `cache_key`: cache key being checked
- `is_stale`: boolean
- `contradiction_rate`: 0.0–1.0 (contradictions / total facts checked)
- `stale_reason`: optional explanation if stale
- `sources_checked`: list of source identifiers
- `contradictions_found`: list of specific contradictions
- `metadata`: `ArtifactMetadata`

Staleness thresholds:
- Content is stale if `contradiction_rate >= 0.10` (configurable via `STALENESS_CONTRADICTION_THRESHOLD`)

---

## Validation Best Practices (Operational)
1. Validate at every boundary:
   - user → Node (Zod)
   - Node → Python request (Zod)
   - Python → Node response (AJV)
2. Log all validation failures with:
   - `request_id`, `schema_version`, validation error list, retry_count (if applicable)
3. Never silently coerce or "fix" invalid data.
